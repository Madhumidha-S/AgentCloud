# AgentCloud

AgentCloud is a lightweight, CPU-efficient, multi-agent system that simulates a cloud environment and autonomously detects, diagnoses, and mitigates system incidents using LLM-based reasoning and agentic workflows. 

At its core, this project demonstrates a modern **ReAct (Reason + Act)** loop for autonomous server and incident response without requiring a GPU or massive external dependencies.

---

## System Architecture

The project is structured into a sequential agentic pipeline that operates continuously:

`Log Generator → Monitoring → Diagnosis → Planning → Execution → Memory`

1. **Simulator (Log Generator)**: Generates real-time mock cloud metric logs (e.g. CPU spikes, authentication failures, segfaults). Available scenarios: `normal`, `overload`, `crash`, `intrusion`, and `mixed`.
2. **Monitoring Agent**: Anomaly detection logic. Scans continuous streaming logs using time-windowed thresholds to instantly alert on anomalous behavior.
3. **Diagnosis Agent**: Uses an LLM (with a pure CPU-fallback) to classify the anomaly (root cause & severity). Guarantees JSON schema constraint formatting dynamically.
4. **Planning Agent**: Decides the mitigation route (`restart_service`, `block_ip`, `isolate_server`, `reroute_traffic`). Actively pulls context from Memory to reuse successful strategies or dodge failed ones.
5. **Execution Agent**: Simulates the system commands internally. Exposes success/failure states back to the pipeline.
6. **Memory Agent**: Powered by an embedded SQLite database. Stores a long-term ledger of incidents and actions, allowing the system to demonstrate "Failure -> Recovery" learning over multiple iterations.

## Installation & Setup

AgentCloud can be run strictly using standard Python libraries.

```bash
# Clone the repository
git clone https://github.com/Madhumidha-S/AgentCloud.git
cd AgentCloud

# Set up a virtual environment
python3 -m venv .venv
source .venv/bin/activate

# Install requirements
pip install -r requirements.txt
```

## Quickstart

Run a completely simulated autonomous cycle using the default CPU deterministic fallback (no LLM required):

```bash
python3 -m agentcloud.main --scenario mixed --max-events 5
```

Live simulator logs will be written to `logs/cloud.jsonl` in JSONL format. Long-term semantic memory and past resolutions are stored in `database.db` (SQLite).

If you want to view the raw JSON line logs in real-time as they are generated by the simulator, add the `--show-logs` flag:
```bash
python3 -m agentcloud.main --scenario mixed --max-events 5 --show-logs
```

### Supported Scenarios
You can run the system to test specific failure cases by changing the `--scenario` flag. The simulator supports:
- `normal`: Standard cloud operation with fluctuating metrics
- `overload`: Massive CPU and traffic spikes triggering node load anomalies
- `intrusion`: Coordinated IP brute-force and failed auth bursts
- `crash`: Unexpected service segmentation faults and offline nodes
- `mixed`: (Default) Rotates randomly through all of the above

## Hybrid AI & ML Integration

AgentCloud now utilizes a highly scalable **Hybrid AI Architecture**. 

Instead of routing every single log to a slow Generative LLM, the `DiagnosisAgent` uses a fine-tuned **DistilBERT** model (`agentcloud/models/cloudsec-model`) to classify anomalies with near-zero latency. 
If the ML model detects an anomaly with high confidence (>= 0.40), it instantly mitigates the issue. If the model is unsure, it gracefully falls back to the Generative LLM for deep reasoning.

### Training the ML Classifier
You can locally fine-tune the DistilBERT model on your own incident data:
```bash
# Ensure you have installed the PyTorch/HuggingFace dependencies
pip install -r requirements.txt

# Run the training script (Uses agentcloud/data/train.jsonl)
python3 train_model.py
```

### Option A: Local LLM Fallback (Qwen GGUF)
*Recommended for fully private, API-free generative fallbacks.*

```bash
# 1. Download the GGUF model
mkdir -p models
curl -L -o models/qwen2.5-0.5b-instruct-q4_k_m.gguf "https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct-GGUF/resolve/main/qwen2.5-0.5b-instruct-q4_k_m.gguf"

# 2. Increase context window
export AGENTCLOUD_LOCAL_MODEL_CTX="4096"
export AGENTCLOUD_LOCAL_MODEL_PATH="models/qwen2.5-0.5b-instruct-q4_k_m.gguf"

# 4. Run the demo
python3 -m agentcloud.main --scenario overload --max-events 5
```

### Option B: OpenAI API
If you prefer a heavier reasoning engine (like `gpt-4o-mini`):

```bash
export OPENAI_API_KEY="sk-..."
export AGENTCLOUD_LLM_MODEL="gpt-4o-mini"
python3 -m agentcloud.main --scenario mixed --max-events 5
```

## Evaluation & Ablations

AgentCloud has built-in features for research and rigorous reporting:

### Memory Learning Demo
Forces an incident to occur, simulates a failure using a bad mitigation path, stores that failure, and successfully resolves the incident the second time around by referencing past failures.

```bash
python3 -m agentcloud.main --demo-learning --tick 0.02 --trace-file logs/trace.jsonl
```

### Ablation Studies
Turn off specific cognitive components to calculate success differentials:
```bash
# Without the Memory Agent (cannot recall past successes/failures)
python3 -m agentcloud.main --scenario mixed --max-events 5 --no-memory

# Without the Planning Agent (defaults to an imperative fallback map)
python3 -m agentcloud.main --scenario mixed --max-events 5 --no-planning
```

### Evaluation Runner (Metrics)
Run massive scenario blocks across episodes to automatically calculate **Success Rate**, **Tool Accuracy**, and **Mean Time To Recovery (MTTR)**:

```bash
# Run baseline evaluations
python3 -m agentcloud.evaluation --scenario mixed --episodes 10 --max-incidents 5

# Compare against ablations
python3 -m agentcloud.evaluation --scenario mixed --episodes 10 --max-incidents 5 --no-memory
python3 -m agentcloud.evaluation --scenario mixed --episodes 10 --max-incidents 5 --no-planning
```

## Project Outputs
- **Logs**: Continuous logs trace out to `logs/cloud.jsonl`.
- **Memory Database**: Stored autonomously generated incidents are recorded permanently in `database.db`.
- **Trace Output**: Run any `main.py` loop with `--trace-file logs/trace.jsonl` to append extremely verbose JSON trace records for the agentic thought-loop.
